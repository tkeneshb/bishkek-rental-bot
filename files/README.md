# Bishkek Rental Bot

Telegram-бот, который собирает объявления о квартирах в Бишкеке с
House.kg и Lalafo, отфильтровывает риелторов и присылает покупателям
только то, что выкладывают сами хозяева.

Я делал его, потому что устал — каждое второе объявление на этих
сайтах от агентства, и фильтр "только собственник" там не работает.
Хотелось проверить, можно ли вытащить настоящих хозяев из этого
потока, и заодно попробовать LLM на реальной задаче, а не на учебной.

## Что внутри

Парсер ходит по House.kg и Lalafo раз в N минут, забирает новые
объявления, складывает в SQLite. Дальше каждое объявление проходит
через семь слоёв проверок — что-то блокирует сразу, что-то отправляет
в очередь модерации:

1. **Claude** читает текст и решает, кто это писал — собственник или
   риелтор. Возвращает `is_owner`, `confidence`, `reason`. Если
   уверенность ≥ 0.85 и это риелтор — блок. Между 0.5 и 0.85 — в
   очередь к модератору, потому что ложно заблокировать настоящего
   хозяина хуже, чем пропустить риелтора.
2. **Лимит на аккаунт** — максимум 2 активных объявления с одного
   Telegram-пользователя.
3. **Кулдаун 24 часа** между объявлениями с одного аккаунта.
4. **Дедуп по телефону** — один номер = одно объявление. Номера
   нормализуются к виду `996XXXXXXXXX` перед сравнением.
5. **pHash на фото** — реализовал DCT-хэширование сам, через Pillow.
   Картинка ужимается до 32×32 в grayscale, применяется 2D DCT,
   берётся верхний левый угол 8×8 — это даёт 64-битный отпечаток.
   Сравнение через расстояние Хэмминга, порог 8. Дистанция ≤ 3 —
   жёсткий блок (явно те же фото), 4–8 — в модерацию. DCT устойчив к
   ресайзу, сжатию и изменению яркости, поэтому риелторы не могут
   просто пережать картинку.
6. **Жалобы покупателей** — кнопка "это риелтор" под каждым
   объявлением. После 3 жалоб уходит в очередь автоматически.
7. **Ручная модерация** — `/modqueue` для админов, где висят все
   спорные случаи.

Покупатели подписываются на район + диапазон цен и комнат, бот
присылает новые объявления в личку как только они проходят все
проверки.

## Стек

- Python 3.11, asyncio
- `python-telegram-bot` для бота
- `anthropic` — Claude Haiku для классификации (быстрая и дешёвая)
- `httpx` + `BeautifulSoup` — парсинг
- `Pillow` — pHash (DCT написан вручную, без `imagehash`)
- SQLite с WAL-режимом

Никаких тяжёлых зависимостей вроде PyTorch или OpenCV нет
специально — хотел уложиться в дешёвый VPS.

## Запуск

```bash
cp .env.example .env
# впиши свои TELEGRAM_TOKEN и ANTHROPIC_API_KEY

pip install -r requirements.txt
python bot.py
```

Парсер запускается отдельно, не из бота:

```bash
python -c "import asyncio; from parser import run_pipeline; asyncio.run(run_pipeline(pages=3))"
```

Обычно я запускаю его по cron каждые 30 минут.

## Структура

```
bot.py          — Telegram-бот, все хэндлеры команд и состояния диалогов
parser.py       — парсеры House.kg и Lalafo + полный пайплайн
defense.py      — семь слоёв проверок, включая обращение к Claude
photo_guard.py  — pHash через Pillow и DCT, расстояние Хэмминга
database.py     — схема SQLite и все запросы к БД
```

## Что работает плохо

- Селекторы парсеров на CSS-классах — House.kg и Lalafo иногда
  меняют вёрстку, и парсер тихо начинает возвращать 0 объявлений.
  Нужен мониторинг "сколько объявлений мы добыли за последний час".
- Claude иногда вешает риелторский ярлык на собственников, которые
  пишут слишком сухо и по делу. Промпт переписывал три раза, всё
  ещё не идеально. Возможно, надо обучать на собранных примерах,
  но это уже отдельная задача.
- SQLite держит нагрузку, но при росте до тысяч объявлений в день
  упрётся в блокировки на запись. Надо будет переезжать на
  PostgreSQL.
- pHash хорошо ловит идентичные фото, но не понимает, что это
  "та же квартира с другого ракурса". Тут нужна уже модель
  компьютерного зрения, а не хэш.

## Почему я это делал

Магистерская диссертация у меня по отказоустойчивому планированию
распределённых задач — это академическая работа, в продакшене её
никто не видит. А этот бот — наоборот, маленький, но реальный: его
кто-то использует, и видны живые ошибки. Хотел понять, как
ощущается разрыв между "система работает на тестах" и "система
работает у людей". Оказалось, что технических проблем меньше, чем
организационных: где провести границу между блоком и модерацией,
кто отвечает за ложное срабатывание, как выстроить доверие к
автоматическому решению. Это интереснее самого алгоритма.
